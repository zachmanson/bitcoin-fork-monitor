Create `app/routers/analytics.py` with two new GET endpoints:

**GET /api/analytics/stale-rate-over-time**
- Accepts query param `period: str = "monthly"` (valid values: `"weekly"`, `"monthly"`)
- Queries the Block table grouped by the period (week or calendar month of the block's timestamp)
- For each bucket: count canonical blocks and orphaned blocks, call `calculate_stale_rate()`
- Returns a list sorted by period ascending:
  ```json
  [
    {"period": "2009-01", "canonical": 1234, "orphaned": 5, "stale_rate": 0.004},
    ...
  ]
  ```
- For weekly periods, format as `"2009-W01"` (ISO week). For monthly, format as `"YYYY-MM"`.
- Explanation: SQLite's `strftime('%Y-%m', timestamp)` for monthly and `strftime('%Y-W%W', timestamp)` for weekly — use `func.strftime` from SQLModel/SQLAlchemy.

**GET /api/analytics/era-breakdown**
- No query params
- Each difficulty adjustment era is a 2016-block window: era 0 = heights 0–2015, era 1 = heights 2016–4031, etc.
- Compute era number as `height // 2016`
- For each era: count canonical and orphaned blocks, compute stale rate, record min/max height
- Returns list sorted by era number ascending:
  ```json
  [
    {
      "era": 0,
      "height_start": 0,
      "height_end": 2015,
      "canonical": 2016,
      "orphaned": 0,
      "stale_rate": 0.0,
      "low_confidence": true
    },
    ...
  ]
  ```
- `low_confidence: true` for all eras where `height_start < 321000` (approximately pre-2015; Bitcoin height ~321000 was reached in late 2014 / early 2015)
- Use SQLite integer division: `(Block.height / 2016)` (SQLAlchemy integer column division floors automatically)

Wire the router into `app/main.py`:
- Add `from app.routers import analytics` to the imports
- Add `app.include_router(analytics.router)` after the existing routers

Include module docstrings explaining what the endpoints do and why eras are defined as 2016-block windows (not years or halvings). The developer should understand this is the most technically precise era boundary.

Create `frontend/svelte.config.js`:
```js
// SvelteKit configuration.
// adapter-static builds to a folder of static HTML/JS/CSS files.
// This means no server-side rendering (SSR) — the app is a pure client-side SPA.
// Why static adapter? FastAPI will serve these files in production. Static files
// are simpler to serve than a Node.js SSR server alongside Python.
import adapter from '@sveltejs/adapter-static';

/** @type {import('@sveltejs/kit').Config} */
const config = {
  kit: {
    adapter: adapter({
      // Output goes to frontend/build/ — FastAPI will serve from here
      pages: 'build',
      assets: 'build',
      fallback: 'index.html',   // SPA fallback: any path serves index.html
    }),
  },
};

export default config;
```

Create `frontend/vite.config.ts`:
```ts
// Vite is the build tool that powers SvelteKit's dev server.
// The proxy config here routes any fetch('/api/...') from the browser
// to FastAPI running at localhost:8000 — this avoids CORS errors during
// development without needing to configure CORS headers on the backend.
import { sveltekit } from '@sveltejs/kit/vite';
import { defineConfig } from 'vite';

export default defineConfig({
  plugins: [sveltekit()],
  server: {
    proxy: {
      // Any request to /api/* from the browser is forwarded to FastAPI
      '/api': {
        target: 'http://localhost:8000',
        changeOrigin: true,
      },
    },
  },
});
```
