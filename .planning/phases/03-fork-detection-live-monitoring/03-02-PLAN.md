    WS_URL = "wss://mempool.space/api/v1/ws"

    Implement these functions in order (each depends on the previous):

    `_wait_for_backfill() -> None`: Opens a short-lived Session(engine) every BACKFILL_POLL_INTERVAL_SECONDS, queries SyncState, returns when backfill_complete is True. Log INFO on first call ("Monitor waiting for backfill...") and on return ("Backfill complete — starting live monitor").

    `_process_block(session: Session, block_data: dict, pending_resolutions: list) -> None`:
    - Extract hash (block_data["id"]), height (block_data["height"]), timestamp from Unix int to datetime (datetime.utcfromtimestamp(block_data["timestamp"]))
    - Upsert the Block row: check if Block with this hash exists; if not, insert it with is_canonical=True
    - Call detect_fork_at_height(session, height, new_hash)
    - If fork detected: call fetch_block_status for both hashes to get in_best_chain. Determine canonical vs orphaned. If both return in_best_chain=True (ambiguous), append to pending_resolutions and call write_fork_event with resolution_seconds=None. Otherwise call write_fork_event normally.
    - Retry any pending_resolutions items: for each pending item, re-fetch status for both hashes; if resolvable, update ForkEvent.resolution_seconds and remove from list
    - Update SyncState: get the state row, set last_synced_height = max(state.last_synced_height, height), set updated_at = datetime.utcnow(), commit

    `_rest_gap_fill(session: Session, state: SyncState) -> None`:
    - Get tip = fetch_tip_height()
    - Walk from state.last_synced_height to tip using fetch_blocks_page (same page-walking pattern as backfill.py)
    - Call _process_block for each block at height >= state.last_synced_height, in ascending order
    - Sleep REQUEST_THROTTLE_SECONDS between pages (import from api_client)

    `_ws_loop(session: Session, pending_resolutions: list) -> None`:
    - Connect using `websockets.sync.client.connect(WS_URL, open_timeout=30, ping_interval=30, ping_timeout=10)`
    - Send SUBSCRIBE_MSG immediately after connect
    - Iterate messages: parse JSON, call _process_block if "block" in msg; ignore all other keys
    - Raises ConnectionClosed or Exception on disconnect — caller handles retry

    `run_monitor() -> None`:
    - Call _wait_for_backfill()
    - Create a single Session(engine) to be reused across the monitor lifecycle (not per-block)
    - Initialize pending_resolutions = []
    - consecutive_failures = 0; in_rest_fallback = False; last_ws_attempt = 0
    - Main loop: if not in_rest_fallback, attempt _ws_loop. On success reset consecutive_failures=0. On failure increment consecutive_failures. At WS_FAILURE_THRESHOLD, set in_rest_fallback=True, log WARNING.
    - If in_rest_fallback: run _rest_gap_fill, sleep REST_POLL_INTERVAL_SECONDS. Every WS_RECONNECT_INTERVAL_SECONDS attempt _ws_loop again — if succeeds, set in_rest_fallback=False, log INFO recovery.
    - Any unhandled exception in the outer loop should be logged at ERROR and cause a short sleep before retry (avoid tight crash loop).

    Note for the developer: The state machine here (WebSocket mode → REST fallback → WebSocket recovery) is a common pattern in resilient data pipelines. The key insight is tracking consecutive_failures rather than any failure, so a single blip doesn't trigger fallback.
